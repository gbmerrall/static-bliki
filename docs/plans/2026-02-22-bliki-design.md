# Bliki Design

A static site generator that produces a blog with wiki-style cross-linking.
Single author, public-facing. Written in Python.

## Content Model

Every piece of content is a directory containing an `index.md` and optional assets (images).

```
content/
  posts/
    2026-02-22-my-post/
      index.md
      hero.jpg
    2026-02-20-another-post/
      index.md
  wiki/
    python/
      index.md
      logo.png
    design-patterns/
      index.md
```

### Content Types

- **Posts**: dated, chronological. Front matter includes `date`, `title`, `category`, `summary`,
  `draft`.
- **Wiki pages**: undated, evergreen, topic-based. Front matter includes `title`, `category`,
  `summary`.
- **Categories**: single value per page (not a list). Category index pages are auto-generated
  from whatever categories appear across all content.

### Front Matter (YAML)

```yaml
# Post
---
title: My First Post
category: programming
summary: A short description for the index page
date: 2026-02-22
draft: false
---

# Wiki page
---
title: Python
category: languages
summary: Notes on Python
---
```

### Formatting

Markdown with `[[wiki links]]` syntax via Python-Markdown's wikilinks extension.

- `[[Page Name]]` resolves to the matching post or wiki page URL
- `[[Page Name|display text]]` for custom link text
- Links resolve by matching against page titles in the content registry
- Broken links generate build warnings
- Images use standard markdown syntax with relative paths: `![alt](photo.jpg)`

### Backlinks

Computed at build time. Each page gets a "Pages that link here" section listing every page
that contains a `[[wiki link]]` pointing to it. Stored as an inverted link graph.

## Architecture

Python CLI tool. No web framework. Dependencies:

- `python-markdown` + `markdown.extensions.wikilinks` -- markdown rendering with wiki links
- `jinja2` -- template rendering
- `python-frontmatter` -- YAML front matter parsing
- `pygments` -- syntax highlighting for code blocks
- `pillow` -- image resizing and WebP conversion
- `click` -- CLI interface
- `livereload` -- dev server with file watching and browser auto-refresh
- `pagefind` -- external binary, run post-build for client-side search index

## Build Pipeline

`bliki build` runs these steps in order:

1. **Scan content**: walk `content/posts/` and `content/wiki/`, parse front matter from each
   `index.md`, build an in-memory registry of all pages (title, slug, type, category, path).

2. **Resolve wiki links**: first pass through all markdown to find `[[Page Name]]` references.
   Build a link graph (A links to B). Invert to get backlinks (B is linked from A). Warn on
   broken links.

3. **Render markdown**: Python-Markdown with extensions:
   - `wikilinks` -- custom URL builder that resolves against the page registry
   - `fenced_code` + `codehilite` -- syntax highlighting via Pygments
   - `toc` -- table of contents data for templates

4. **Process images**: for each content directory containing images:
   - Resize via Pillow to configured widths (600, 1200, original)
   - Convert to WebP (keep original format as fallback)
   - Rewrite `<img>` tags in rendered HTML to `<picture>` elements with `srcset`

5. **Render templates**: pass markdown HTML + metadata into Jinja2 templates. Generate:
   - Individual post and wiki pages
   - Post index (chronological, newest first) -- this is the homepage
   - Wiki index (alphabetical)
   - Per-category pages (listing both posts and wiki pages in that category)
   - Category index
   - Search page (Pagefind UI shell)
   - RSS/Atom feed

6. **Copy static assets**: CSS, JS, icons from theme into `output/static/`.

7. **Run Pagefind**: shell out to `pagefind --site ./output` to build the search index.

## Output Structure

```
output/
  index.html                          # Post index (homepage)
  wiki/
    index.html                        # Wiki index
    python/
      index.html
      logo.png / logo-600.webp / logo-1200.webp
  posts/
    2026/02/my-first-post/
      index.html
      hero.jpg / hero-600.webp / hero-1200.webp
  categories/
    index.html
    programming/
      index.html
  static/
    css/ js/ images/
  pagefind/                           # Generated by Pagefind
  feed.xml                            # RSS/Atom feed
```

## Templates

Adapted from the pelican-pickles theme (Jinja2, CSS custom properties, vanilla JS).

```
templates/
  base.html              # Layout shell: head, nav, footer, dark mode toggle
  post.html              # Single blog post
  wiki.html              # Single wiki page (includes backlinks section)
  post_index.html        # Chronological post listing (homepage)
  wiki_index.html        # Alphabetical wiki page listing
  category.html          # Posts + wiki pages for one category
  category_index.html    # List of all categories
  search.html            # Pagefind search page
```

### Kept from pickles

- CSS custom properties for theming
- Dark mode toggle (localStorage persistence)
- Responsive navigation (hamburger on mobile)
- Reading progress bar
- Code block copy button
- Syntax highlighting styles
- Clean typography

### Dropped from pickles

- Disqus comments
- Google Analytics
- Social share buttons
- Pelican-specific template variables

### Added for bliki

- Backlinks section on every page
- Visual distinction between posts (dated) and wiki pages in listings
- Wiki index page
- Category pages spanning both content types

## Image Processing

Automatic at build time. Author workflow: drop a full-size image next to `index.md`, reference
it with standard markdown `![alt](filename.jpg)`.

The build step:
1. Copies original image to output
2. Generates resized variants (600w, 1200w) via Pillow
3. Converts each variant to WebP
4. Rewrites `<img>` in rendered HTML to `<picture>` with `srcset` and format fallbacks

## CLI

```bash
bliki build              # Full build to ./output/
bliki build --drafts     # Include draft posts
bliki serve              # Build + livereload dev server on localhost:8000
bliki new post "Title"   # Scaffold content/posts/YYYY-MM-DD-title/index.md
bliki new wiki "Title"   # Scaffold content/wiki/title/index.md
```

## Search

Pagefind runs as a post-build step. Generates a WASM-based client-side search index.
The search page includes Pagefind's UI components. No server needed.
